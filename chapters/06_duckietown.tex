\chapter{Experimental validation on the Duckietown platform}
\label{ch:duckietown}

The previous chapters established the modeling and computational foundations of games of ordered preference.
This chapter turns that machinery into a concrete experimental setup using DB21J Duckiebots on the
standardized Duckietown road layout \cite{duckietown:2017}.
The goal is not to introduce a different planner, but to close the loop between the receding-horizon solver
and physical robots in a way that preserves the structure of the optimization problem while respecting the
constraints of a real deployment.

The need for an explicit experimental layer follows from the gap between mathematical feasibility and
physical execution.
In simulation, the solver can query the current state, solve a receding-horizon game, and apply the first
control input without worrying about network delays, actuator saturation, watchdog timeouts, or sensor
dropouts.
On real robots, every one of those effects matters.
The purpose of this experimental stack is therefore to preserve the structure of the solution method while
making its execution observable and safe enough for physical trials.

Duckietown is particularly convenient for that purpose because it standardizes the small-scale road geometry,
the DB21J vehicle footprint, and the ROS-based workflow used around the robots.
In this thesis, that standardized environment is used as a validation scaffold for \cref{ch:lex-ibr-ot}: the same
lexicographic IBR over time logic is executed, but the state and command loop is mediated by the
robotics stack shown in \cref{fig:duckietown_architecture}.

This validation setup is not fully decentralized.
The sensing available on our DB21J units is not sufficient for each robot to reliably estimate the position
and velocity of the other vehicles on its own.
In particular, with a simple onboard webcam, detecting and tracking the surrounding robots robustly enough
for closed-loop game execution is not currently realistic.
For that reason, the experiments use a centralized runtime loop that aggregates the state information needed
by the solver before redistributing the resulting commands.

\section{System architecture}
\label{sec:duckietown-architecture}

The setup uses three DB21J Duckiebots connected to a central computer that
coordinates the runtime loop and solves the games, while each robot manages its
own actuation and sensing interface.
This keeps the optimization layer independent from the transport and control
details of the physical platform.\footnote{The code used for the experiments in this chapter is available at \url{https://codeberg.org/ericroy/ordered-preferences-duckietown}. We mainly used the multi-robot interface.}

\Cref{fig:duckietown_architecture} summarizes the closed-loop execution path.
At each sampling instant \(\Delta t\), the runtime manager receives the latest per-robot state estimates
\(\hat{\mathbf{x}}_k^i\) over ROS, bundles them into \(\hat{\mathbf{X}}_k\), and queries the bridge.
The bridge issues the local TCP call to the solver, which replans only after the current open-loop plan has
executed for \(T_l\) steps and otherwise returns the next command from the existing plan.
Each robot-side adapter then maps the returned sample to the low-level drive input
\(u_k^i = (v_k^i,\omega_k^i)\), while the watchdog enforces the zero-input fallback if the command stream is
interrupted.

% If no valid command is returned in time, the watchdog falls back to a zero command rather than leaving a

\begin{algorithm}
\KwData{Solver bridge, sampling period \(\Delta t\), turn length \(T_l\), robot set, goal set}
\KwResult{Closed-loop acceleration commands \(\mathbf{a}_k^\star\)}
\medskip
Initialize the manager--robot communication and the state / command interfaces\;
\While{the experiment is running}{
    Read the latest state estimate \(\mathbf{x}_k\)\;
    Query the bridge for the next control sample; replan only after \(T_l\) executed steps\;
    \If{a valid command is available}{
        Publish the returned acceleration command \(\mathbf{a}_k^\star\)\;
    }
    \ElseIf{a fallback policy is available}{
        Replay the next fallback command\;
    }
    \Else{
        Publish the zero command\;
    }
    Wait until the next tick \(\Delta t\)\;
}
\caption{Closed-loop manager cycle in the DB21J Duckietown validation stack.}
\label{alg:duckietown_manager_loop}
\end{algorithm}

The manager therefore advances the closed loop every \(\Delta t\), while the underlying receding-horizon
game is refreshed only every \(T_l\) steps and otherwise continues along the current plan.
The explicit safety boundary remains the same: until a feasible command is available, the robots remain
stationary (\cref{alg:duckietown_manager_loop}).

\section{Robot-side control and telemetry}
\label{sec:duckietown-control}

The control variable used throughout \cref{ch:goop,ch:lex-ibr-ot} is a planar
acceleration, whereas the DB21J Duckiebot does not accept acceleration commands directly.
Its native interface is the low-level pair \((v,\omega)\), so the solver output must be converted before it
can be executed on the robot.

The robot adapter implements a compact controller that maps the commanded planar acceleration to a
velocity-vector state and then to the standard driving inputs.
Let \(\mathbf{a}_k = (a_{x,k}, a_{y,k})\) denote the acceleration vector received at time step \(k\).
After clamping \(\mathbf{a}_k\) to the admissible magnitude and applying a velocity-decay factor, the
controller updates its internal planar velocity estimate as
\begin{equation}
\mathbf{v}_{k+1} = \alpha \mathbf{v}_k + \Delta t \,\mathrm{clip}_{a_{\max}}(\mathbf{a}_k),
\end{equation}
where \(\alpha = \max(0, 1 - \lambda \Delta t)\) and \(\lambda\) is the decay parameter.
The desired heading is then extracted from the updated vector,
\begin{equation}
\theta^{\mathrm{des}}_{k+1} = \operatorname{atan2}(v_{y,k+1}, v_{x,k+1}),
\end{equation}
and the controller generates the command pair
\begin{align}
\omega^{\mathrm{cmd}}_{k+1}
&=
\mathrm{clip}_{\omega_{\max}}
\!\left(
 k_\psi\,\mathrm{wrap}_\pi\!\left(\theta^{\mathrm{des}}_{k+1} - \theta_k\right)
\right), \\
v^{\mathrm{cmd}}_{k+1}
&=
\mathrm{clip}_{[0,v_{\max}]}\!\left(\lVert \mathbf{v}_{k+1} \rVert\right).
\end{align}
This structure is expressive enough to preserve the qualitative
directional behavior induced by the acceleration command while maintaining simplicity.
At the same time, the optimization problem must remain constrained to accelerations that are compatible with
the physical platform, so that the commanded motion can be mapped to feasible low-level inputs instead of
producing requests that the robot cannot track reliably.

The published \texttt{RobotStateStamped} message is richer than a simple pose estimate.
In addition to \(x\), \(y\), \(\theta\), \(v_x\), and \(v_y\), it carries the command pair
\(v^{\mathrm{cmd}}\) and \(\omega^{\mathrm{cmd}}\), the encoder-based estimates when available, and the
wheel command feedback when the robot firmware exposes it, to allow a more precise position estimate for the next solve step.
The adapter also includes a watchdog timer that zeros the command when no fresh acceleration message has been
received within a configured timeout.

\section{Discussion}

This chapter shows that the acceleration-based receding-horizon game formulation developed in the previous
chapters can be embedded in a physical robotic loop without altering its basic mathematical structure.
The experimental implementation integrates the solver, the centralized runtime manager, and the robot-side
adapter into a coherent closed-loop system on DB21J Duckiebots and standardized Duckietown infrastructure,
thereby providing an experimental validation beyond simulation.

However, the scope of this validation remains limited.
The DB21J sensing stack and the chosen deployment configuration motivate a centralized architecture and a
conservative control stack, and do not support a fully decentralized treatment of multi-robot interaction.
Accordingly, the contribution of this chapter is methodological rather than definitive: it establishes the
practical executability of the proposed formulation under real timing, actuation, and safety constraints,
while a more comprehensive assessment would require richer perception, less centralized execution, and
broader experimental campaigns.
